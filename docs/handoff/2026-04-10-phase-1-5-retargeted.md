However, Suresh's stated goal in his own words is broader: he wants nexus-ui-repair to test all 89 pages of Nexus OS, find what's broken, and produce a loop where the system fixes itself with minimal human intervention, because he is a solo developer and manual repair of 89 pages is not sustainable. A scout that only produces triage queues does not close the loop — Suresh still has to fix every bug the scout finds. This mismatch was surfaced when Suresh expressed exhaustion at the prospect of manually fixing years worth of UI bugs and asked for a brutally honest recommendation.

## Frame reset: scout + Claude Code repair loop

The agreed path forward (Option C from the planning conversation) is:

The scout continues to be read-only. It does not write code, does not apply fixes, does not modify anything. Phase 1.4's governance-first architecture is preserved.

Claude Code becomes the repair half of the loop. Suresh already uses Claude Code as his daily driver for autonomous coding work. Claude Code is a coding agent with write access to the repo, governed by Suresh's rules, with existing HITL approval workflows. It does not need to be rebuilt.

The seam between scout and Claude Code is a structured JSON repair-ticket output format. Phase 1.5 ships the scout with this format as its primary deliverable — the scout produces one JSON file per run containing structured repair tickets that Claude Code can consume directly. Each ticket contains: finding id, page and sub-view location, DOM context, error strings verbatim, severity, screenshot path, and enough context for Claude Code to identify the relevant component file and propose a fix without human explanation.

The morning workflow becomes: Suresh runs nexus-ui-repair against a page, reads the JSON ticket file, opens Claude Code and says "read the ticket file, pick the highest-severity ticket, propose a fix, show me the diff, wait for approval." Suresh approves or rejects, Claude Code lands the fix or tries a different approach, loop repeats. This is the realistic path to "production grade 89 pages" for a solo developer.

A dedicated autonomous repair agent (as a separate crate, governed with its own capability ACLs and rollback safety) remains the long-term destination but is deferred to a later phase. It will be built on top of the proven scout + Claude Code loop, not instead of it.

## Phase 1.5 shape agreed in planning (not yet drafted as Claude Code prompt)

The shape walkthrough was discussed in planning conversation but the actual Phase 1.5 Claude Code prompt has not yet been written. Key decisions made that should survive into the prompt draft:

- Multi-crate scope approved. Phase 1.5 touches both nexus-ui-repair and web-builder-agent. The web-builder-agent addition is a new Tauri command that exposes structured interactive element data for any page, not just Builder — the command is general and reusable for future scout targets beyond Chat.
- LiveEnumerator as a sibling of the existing fixture Enumerator. New file crates/nexus-ui-repair/src/specialists/live_enumerator.rs that queries the DOM via the new Tauri command. Phase 1.2 fixture Enumerator stays untouched. Sibling pattern, not generalization.
- Ground truth markdown-to-JSONL parser. New module crates/nexus-ui-repair/src/ground_truth/parser.rs that reads the sealed .md doc and produces an in-memory Vec<GroundTruthEntry>. The .md remains the single source of truth, the JSONL is derived at scout startup, never committed.
- Three-way comparison harness with honest F1 semantics. confirmed_match, unknown_new, confirmed_miss buckets. F1 computed only over confirmed_match vs confirmed_miss. unknown_new findings are reported separately as "human triage required" and do NOT count as false positives, because penalizing the scout for finding bugs the human missed would invert the metric. Approved in planning.
- JSON repair-ticket output format as the Phase 1.5 deliverable. This is the new seam to Claude Code. Each ticket contains finding id, page, sub_view, dom_context, error_strings, severity, screenshot_path, suggested_fix_category, component_file_hint, and enough context for Claude Code to act without human explanation. Format should be versioned so future scout versions can evolve the schema.
- Halt handling. C4 introduced DriverOutcome.halt: Option<HaltReason>. Phase 1.5 comparison harness must handle halt case as a first-class outcome — partial runs produce partial reports with a halt_reason field and a partial=true flag.
- Five stop gates. SG1 routing table re-verification, SG2 chrono dep check, SG3 screenshot capture API check on EyesAndHands, SG4 cost ceiling persistence path check, SG5 new: Tauri command surface discovery on web-builder-agent (confirm the existing command registration pattern so the new command matches house style). Halt and report on any failure.
- Three-group checkpointed structure (same pattern as Phase 1.4). Group A is ground truth parser + comparison harness + JSON ticket schema (pure data, testable in isolation). Group B is the new Tauri command on web-builder-agent + LiveEnumerator + driver live-mode config (multi-crate, the biggest structural change). Group C is end-to-end integration test against the real Chat page plus the JSON ticket generator running after the scout completes. Each group ends with cargo fmt + clippy + test green and halt-and-report summary.

## Notes for any future Claude session reading this doc

The fabricated Teams ground truth doc episode revealed a failure mode worth internalizing: sealed documents in docs/qa/ or docs/handoff/ may have been generated by previous Claude sessions without being verified against reality. Do not treat any sealed document as authoritative until you have spot-checked at least one concrete claim against the actual codebase or against Suresh's direct confirmation. The seal discipline is a tool for calibration, not a substitute for verification.

The "process notes" sections in the April 8 and April 9 handoff docs were written in Suresh's voice but not by Suresh. Quote-attributing anything to Suresh from those sections is a mistake. If you need Suresh's actual opinions, ask him directly.

Crate #69 as built is a scout, not a repair agent. If you are designing a phase for nexus-ui-repair and your design assumes it can write code, modify files, or apply fixes autonomously, you are building the wrong thing. Fixes come from Claude Code consuming the scout's JSON tickets, with Suresh approving each one. That architectural separation is intentional and preserves the governance-first principle of the rest of Nexus OS.
