Prerequisites

You need three fundamentals before the stack comes alive locally.

  • Rust 1.75+ for building the kernel and workspace crates.
  • Node.js 18+ for the Tauri desktop frontend and app tooling.
  • Ollama for the local LLM runtime behind cognition and vision loops.

Optional but recommended: Docker for deployment, Playwright-compatible browser tooling, and Git for cloning and updates.

Installation

There is no build tool ceremony here. Clone, compile, install app dependencies, then launch.

install.sh
$ git clone https://gitlab.com/nexaiceo/nexus-os
$ cd nexus-os
$ cargo build --workspace
$ ollama pull qwen3.5:4b
$ ollama serve
$ cd app && npm install && npm run tauri dev

Your first agent

Start with a safe L1 or L2 agent so you can learn the shape of the system without touching high-risk autonomy.

  1. Open the desktop app and enter the Chat workspace.
  2. Select Nexus CodeSentry if you want a read-only scan, or Nexus Sentinel if you want a governed security review.
  3. Send a direct goal such as “Scan the kernel for code quality issues and rank them by severity.”
  4. Watch the loop: perceive, reason, plan, act, reflect, learn.
  5. Review the audit trail and approval prompts if the agent approaches a risky boundary.
chat prompt
# Example first message
Scan the kernel/src directory for code quality issues and produce a severity-ranked report.

Configuration reference

Nexus OS is designed to be inspectable. Environment-level configuration keeps the runtime predictable.

# .env OLLAMA_HOST=http://localhost:11434 NEXUS_DB_PATH=./data/nexus.db NEXUS_LOG_LEVEL=info NEXUS_FUEL_DEFAULT=10000 NEXUS_AUDIT_DIR=./data/audit NEXUS_APPROVAL_MODE=interactive

Common knobs include default autonomy levels, audit paths, model endpoints, and approval routing behavior.

Agent manifest schema

Every agent is defined by a manifest that names the role, its resource budget, allowed capabilities, and governance requirements.

[agent] id = "nexus-example" name = "Nexus Example" version = "1.0.0" description = "What this agent does" autonomy_level = 2 [resources] fuel_budget = 10000 model = "qwen3.5:9b" [capabilities] allowed = ["fs.read", "fs.write", "process.exec"] [schedule] cron = "0 */4 * * *" [governance] hitl_tier = "Tier1" risk_level = "medium"

API reference overview

The API layer mirrors the platform architecture: agents, approvals, memory, audit logs, schedules, and runtime state.

  • Agents API for listing, starting, stopping, and inspecting agents.
  • Approval API for queueing, approving, or rejecting risk-bound actions.
  • Audit API for browsing the immutable event history.
  • Memory API for reading or erasing structured memories.
  • Runtime API for status, metrics, and operator dashboards.

JWT authentication and policy enforcement remain active at the transport layer, not just in the UI.

Full docs and next steps

After the first local run, most teams move into one of four tracks:

  • Customize an existing agent manifest and tune its capability envelope.
  • Turn on a scheduled agent like Sentinel, Phantom, or Sovereign.
  • Explore the Approval Center and audit chain to understand production controls.
  • Deploy with Docker and connect the messaging bridge for operational workflows.
