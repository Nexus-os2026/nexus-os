PROCEDURE

1. Open {{URL}} in Chrome. If localhost permission is required, request it once and proceed. Wait for the page to fully load (network idle, no spinners).

2. For each viewport in [1920x1080, 1280x800, 1024x768], in order:
   a. Resize viewport using setDeviceMetricsOverride or equivalent.
   b. Wait 300ms for layout to settle.
   c. Measure horizontal overflow at TWO levels:
      - **User-visible level (P1 candidates):** document.documentElement and document.body. Any overflow here means the page itself scrolls horizontally and the user sees it. ALWAYS report.
      - **Internal element level (classify before reporting):** any element where scrollWidth > clientWidth. For each such element, walk the ancestor chain and check whether any ancestor has computed `overflow`, `overflow-x`, or `overflow-y` set to `hidden`, `clip`, `scroll`, or `auto`. Classify as:
        * **user-visible:** no clipping ancestor found OR documentElement/body itself overflows. The user can scroll to see the bleed.
        * **internal (cosmetic):** at least one clipping ancestor exists and documentElement/body do NOT overflow. The bleed is contained inside a clip box and is invisible to the user.
   d. Identify the main content container by selector and report its scrollWidth/clientWidth alongside documentElement and body.
   You do NOT need to take screenshots in this step. Just measure, classify, and record numbers.

   **Severity rule for overflow findings:**
   - documentElement or body overflow → P1 (user sees horizontal scroll)
   - Internal element overflow classified user-visible → P1
   - Internal element overflow classified internal (cosmetic) → P3 (informational only — by-design decorative bleed inside clip containers, do not surface as P1 unless the visual is genuinely broken)

3. After viewport sweep, return to 1920x1080 for remaining steps.

4. Capture ALL browser console messages from page load. Report every error, warning, log, info, and debug message verbatim with file:line. Do NOT filter by perceived importance — severity grading happens in Findings, not in Console.

5. List every interactive element in the MAIN CONTENT AREA only (exclude sidebar nav `aside.nexus-sidebar-shell`). Element types: button, input, select, textarea, anchor with href. For each: visible label, type, href if any, enabled state.

6. Click each element from step 5, in tab order (top-to-bottom, left-to-right). RULES:
   - Maximum 10 clicks. If more than 10 elements exist, click the first 10 in tab order and report the total count.
   - SKIP any element whose visible label or aria-label contains: "delete", "remove", "destroy", "reset", "wipe", "clear all", "logout", "sign out", "uninstall", "drop". Report skipped elements separately with reason "skipped: destructive".
   - BEFORE each click, record window.location.pathname AND install a MutationObserver that captures every added node and characterData change on document.body for the next 1500ms. Also snapshot document.body.innerText hashed (length + first 200 chars) as a fallback.
   - CLICK the element.
   - WAIT 1500ms (do not poll faster — transient toast badges, error banners, and feedback elements may mount+unmount within this window; the observer must capture them even if they have already cleared by the time you read the result).
   - After the wait, disconnect the observer and read its recorded mutations. For each added node, capture: tagName, className, id, textContent (first 120 chars), and whether the node is still present in the DOM or has already been removed (transient).
   - For each click, capture ALL of these fields (no merging, no skipping):
     * Pathname before
     * New console messages since the last click
     * Failed network requests (4xx/5xx)
     * Visible change: persistent DOM diff (what is different NOW vs before the click)
     * Feedback: transient DOM additions captured by the observer (nodes added then removed within the 1500ms window, OR nodes still present that were not there before). Report every added element even if visually small — toast badges, error banners, inline messages, status indicators.
     * innerText change: yes|no based on the hashed snapshot comparison
     * Pathname after
   - A button is "dead" ONLY if ALL of the following are true: pathname unchanged, zero new console messages, zero failed network requests, zero persistent DOM diff, zero transient DOM additions from the observer, AND no innerText change. If ANY of those show activity, the button produced feedback and must NOT be classified as dead. When in doubt, report the captured feedback under the click result and let the findings pass judge severity — do not pre-classify as dead in the click section.
   - If pathname changed, the click caused navigation. Navigate back via history.back() or by re-opening {{URL}}. Verify pathname returned before the next click.

7. Basic accessibility check (DOM-based):
   - Images without alt: count + up to 5 example selectors
   - Inputs without label or aria-label: count + up to 5 selectors
   - Buttons without accessible name: count + up to 5 selectors
